Composition · one command, many skills

Anatomy of a stack: /ship-feature

One command the human types unfolds into a self-driving loop that ships an entire PRD — every child issue implemented, reviewed, and merged without a human in the loop. It's not one big skill; it's a thin orchestrator composing small, single-purpose skills through two cheap subagents. Here's the whole stack, then the loop it runs.

The stack — control flows top → down

Each layer only knows the layer directly beneath it. The orchestrator never touches code; the agents never decide policy; the skills never loop. Separation is the whole trick.

Why it composes — the load-bearing invariants

Never halt
Every unresolvable condition becomes a cleanup-issue entry, not a prompt. That single rule is what makes autonomy possible.
Cheap subagents
Orchestrator + agents all run on the profile's workhorse model. The pattern assumes subagents are disposable, so it spawns them freely.
Independent review
The reviewer is a separate agent with cold context per PR — it can't be talked into approving its own code, because it isn't the coder.
Isolated worktrees
All work happens in sibling git worktrees. Your main checkout stays yours for the whole run.

Same machine, smaller job

The /ship-issue reuse

/ship-issue <n> is the same orchestrator, same agents, same loop — pointed at a single standalone issue instead of a PRD's children. It drops the outer child-loop and the PRD→master finalize, branches straight off the default branch, and runs the skills in --single mode. One machine, two entry points: that reuse is the payoff of keeping every skill single-purpose.
