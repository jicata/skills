The golden path · idea → merged on master

One feature, the whole SDLC

No single skill ships a feature. The library is a pipeline — each skill takes the artifact the last one produced and hands the next a sharper one: a foggy idea becomes a PRD, a PRD becomes sliced issues, issues become plans, plans become merged PRs. Here is one feature — “saved item collections” — walking the entire path, human-led at the top, autonomous at the bottom.

Ideate→ Specify→ Slice→ Plan→ Build→ Ship
0
Ideate — chart the fog you drive

Find the way, or sharpen the plan

A loose idea that's too big for one session → /wayfinder charts it as a GitHub map of decision tickets and resolves them one at a time until the route is clear. A plan that fits in one session → /grill-me stress-tests it; /grill-with-docs also hardens the glossary & ADRs as decisions land.

/wayfinder /grill-me /grill-with-docs
exampleDecided: collections are per-user, not per-tenant; sharing is a read-only link. Glossary gains the term “Collection.” No code yet — just a settled destination.
1
Specify you drive

Turn the destination into a PRD

/write-a-prd runs a structured interview, explores the codebase, and does the module design up front — then files the PRD as a GitHub issue. The hard boundaries are decided here, on paper, not discovered mid-build.

/write-a-prd
produces PRD #142 · “Saved item collections”
examplePRD #142 filed: problem statement, module design (a Collections slice), out-of-scope list, and the user stories it must satisfy.
2
Slice you drive

Cut the PRD into grabbable slices

/prd-to-issues breaks the PRD into independently-shippable child issues as tracer-bullet vertical slices — each a thin end-to-end cut, not a horizontal layer — wires the Blocked-by edges into a DAG, and prints the execution order.

/prd-to-issues
PRD #142→ #143 create & list #144 add / remove #145 rename / delete #146 share link
exampleExecution order printed: #143 (walking skeleton) → then #144 + #145 in parallel → #146 (blocked by #144).
3
Plan each slice you + agent · HITL

Expand every issue into a real plan

For each child, /expand-issue loads the gospel — the design docs & golden fixtures the issue hard-links, plus the real code — proposes a concrete approach through the repo's coder lens, and resolves genuine decisions with you one at a time. It plans only; it writes no production code. Run it ahead of the build: expand the whole set while the first slice is being implemented.

/expand-issue× each child
example#143 expanded: the table shape, the endpoint contract, and the exact red→green test list are agreed and written back onto the issue — so the builder inherits a plan, not a puzzle.
4
Build — the fork your call, per slice

Two lanes to a merged PR

Every slice reaches the same place — merged into the PRD's base branch — by one of two lanes. You choose per slice.

Lane A · HITL — you drive
Hand-drive a slice to validate the plan by eye. The five-skill loop:
/execute-issue→ /review-pr→ /address-pr→ /merge-pr
Lane B · Autonomous — runs itself
Hand the rest to the orchestrator: it loops coder + reviewer over every remaining child, unattended.
/ship-feature 142
The realistic move — a hybrid. Hand-drive the tracer-bullet slice #143 through Lane A to prove the walking skeleton and the plan hold up. Then let /ship-feature 142 take #144–#146 autonomously — the pattern is proven, so the rest doesn't need your eyes on every round.
▸ how Lane B works inside → the /ship-feature stack · the coder / reviewer split
example#143 reviewed & merged by hand (2 review rounds). #144–#146 handed to /ship-feature — three PRs built, reviewed, and merged to the base branch without a human in the loop.
5
Ship & finalize autonomous

The feature lands on master

When every child is merged, /ship-feature merges the PRD base branch → master and closes the PRD. Nothing ever halted for a human: every unresolvable concern was logged to a single [ship-cleanup] issue instead of stopping the run.

produces feature on master ✓ [ship-cleanup] #150
exampleSaved item collections is live on master. Cleanup issue #150 holds two conceded Axis-B nits — real debt, logged and visible, not silently merged away.

The loop never really ends

Shipping isn't the terminus — three on-ramps feed the same machine.
A bug turns up
/log-issue writes a fully-formed issue → /ship-issue <n> ships it: the same orchestrator + agents as ship-feature, pointed at one issue instead of a PRD.
Cleanup accrued
/drain-cleanup re-verifies each entry in #150 against the current master and spins off well-formed follow-up issues — which re-enter at /ship-issue.
The repo taught you something
A scar learned during the build lands in project-profile.md; /skill-sync flows the generic half back to the base library, so the next repo inherits it for free.
Golden path: /wayfinder · /grill-me → /write-a-prd → /prd-to-issues → /expand-issue → (Lane A: /execute-issue → /review-pr → /address-pr → /merge-pr · Lane B: /ship-feature) → finalize. The example feature is illustrative; issue numbers are for the walkthrough.
