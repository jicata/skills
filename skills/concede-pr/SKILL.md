# Concede PR

Concede a PR review when the orchestrator determines that the maximum number of review iterations has been reached and only `[AXIS-B]` (Code Standards) threads remain. This skill creates a tech debt issue summarizing the architectural compromises, resolves the remaining threads, and approves the PR.

## Invocation

`/concede-pr <pr-number>`

If no number is given, ask the user for one. Do not guess.

## Process

### Step 1 — Fetch PR context and unresolved threads

Run these in parallel:

```bash
gh repo view --json owner,name
gh pr view <n> --json number,title,url
```

Query review threads via GraphQL to find unresolved threads authored by the skill (starting with `Claude comment 🤖`):

```bash
gh api graphql -f query='
query($owner: String!, $repo: String!, $pr: Int!) {
  repository(owner: $owner, name: $repo) {
    pullRequest(number: $pr) {
      reviewThreads(first: 100) {
        nodes {
          id
          isResolved
          comments(first: 20) {
            nodes {
              body
              path
              line
              author { login }
            }
          }
        }
      }
    }
  }
}' -f owner=<owner> -f repo=<repo> -F pr=<n>
```

Filter the threads to those that are `isResolved == false` and where the first comment starts with `Claude comment 🤖` and contains `[AXIS-B]`.

If you find any unresolved threads containing `[AXIS-A]`, **STOP IMMEDIATELY**. Report to the user that Axis A (Requirements) failures cannot be conceded.

### Step 2 — Create the Tech Debt Issue

Read the bodies of the unresolved `[AXIS-B]` threads. Synthesize them into a coherent summary of the architectural compromises or standards violations being deferred.

Create a new issue:

```bash
gh issue create \
  --title "Tech Debt: Address standards violations from PR #<n>" \
  --body "This issue tracks deferred technical debt from PR #<n> (<pr-url>).

The following architectural standards or code quality rules were compromised to unblock the PR:

<Synthesized summary of the unresolved [AXIS-B] threads, including file paths and the specific rules violated>

This was automatically generated by the review orchestrator after reaching the maximum review iteration limit."
```

Capture the URL or number of the newly created issue.

### Step 3 — Resolve the threads

For each unresolved `[AXIS-B]` thread you identified in Step 1, resolve it and leave a reply linking to the new tech debt issue:

```bash
gh api /repos/<owner>/<repo>/pulls/<n>/comments/<comment-id>/replies \
  --method POST \
  -f body="Claude comment 🤖

⚠️ Conceded: This standards violation is being deferred to unblock the PR. Tracked in <new-issue-url>."
```

Then resolve the thread:

```bash
gh api graphql -f query='
mutation($id: ID!) {
  resolveReviewThread(input: {threadId: $id}) {
    thread { isResolved }
  }
}' -f id=<thread-id>
```

### Step 4 — Approve the PR

Submit an `APPROVE` review to unblock the PR:

```bash
gh pr review <n> --approve --body "Claude comment 🤖

⚠️ **Review Conceded**

Maximum review iterations reached. All remaining \`[AXIS-B]\` standards violations have been deferred and are tracked in <new-issue-url>.

Ready to merge."
```

### Step 5 — Report back to the user

Output a concise summary in chat:
- Link to the PR
- Link to the newly created Tech Debt issue
- Number of threads conceded and resolved

## Critical Rules

1. **Never concede an `[AXIS-A]` thread.** If a product requirement is missing, the PR must remain blocked.
2. **Always create the tech debt issue first.** Do not approve the PR without successfully creating the tracking issue.
3. **Always reply to the threads before resolving them.** This leaves a clear audit trail of *why* the thread was resolved despite the code not changing.
