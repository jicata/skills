# Expand Issue

Turn a terse child issue into a fully-planned, **human-agreed** implementation approach — without writing production code. This is the **HITL planning gate**: you + the user nail the *how*; `/execute-issue` (or the afk-coder) then implements it and `/review-pr` reviews it. The agreed plan is persisted **onto the issue** so any agent — a fresh one, a different model, the autonomous coder — can pick it up and build to the same spec.

## When to use
- After `/prd-to-issues`, before `/execute-issue`, when the user wants to plan an issue before handing implementation + review to the agents.
- Especially for issues carrying a real decision, a spike, or architectural ambiguity (HITL issues). Trivially-determined issues can skip straight to `/execute-issue`.

## Principles (read before starting)
- **Gospel first — non-negotiable.** The issue hard-links to its design-doc section(s) + golden fixture(s) (per the PRD's source-of-truth mandate). **Read them. Read the actual code the slice touches.** Never plan from the issue summary alone — that is the #1 failure mode of a fresh agent on a foundation issue.
- **Propose, don't interrogate.** You (the agent) did the homework, so come with a concrete proposal and surface *only* the genuine decisions for the user. This is the inverse of `/grill-me` (which is for when the *user* holds the vision); here you hold most of the answers, and the user's leverage is the few real forks + catching mistakes. (The user is welcome to reverse-grill your proposal.)
- **Teach for maintainership — zoom out, then zoom in.** The exit bar is NOT "the user approved"; it's **"the user could now review this PR — or maintain the slice — as if they'd written it."** That takes more than a one-line rationale per decision; it takes a real senior-to-senior *briefing* (see Step 2.5): the slice's place in the system, a traced worked example, the key abstractions and *why they're shaped that way*, the sharp edges a reviewer must catch, and the change-impact model. Pitch it peer-level — assume the CS fundamentals, spend the words on *this system's* specifics. Ground every claim in real code/the gospel, never abstractions alone. The two questions you must leave fully answered: *"why does this even exist?"* and *"what breaks if I change it?"* Never present a "what" without its "why," and treat "I don't fully get it yet" as the signal to go deeper, not move on.
- **One decision at a time.** Walk the decision tree; resolve each fork before opening the next.
- **Evidence-first for spikes.** If there's an adopt-vs-build / architectural call, scope a spike, get the user's approval on the scope, run it, and decide on the *results* — never by reputation or vibes.
- **Honor the gospel.** The design-doc invariants (and, in brownfield repos with a declared legacy oracle — see `.claude/doctrine/project-profile.md` — its parity gates) are non-negotiable. The plan must conform to the repo's composite coder lens (generated by setup) and the base `tdd` skill (red-green through public interfaces, mock only boundaries).
- **File deferred-scope discoveries to the PRD's catch-all, don't bury them.** Gospel-reading sometimes surfaces something genuinely outside the current scope — a deliberate deferral the gospel already calls out (or should), not a decision you're inventing. Don't silently drop it, silently build past it, or leave it as an orphaned code comment. Search open issues for the PRD's deferred-work catch-all (`PRD #<n> backlog — deferred work catch-all`); if it exists, append an entry (what it is, where discovered, why it's deferred, cross-linked to the doc that says so); if it doesn't, create it with that title pattern. This is distinct from a blocking catch-up issue (drift that must resolve before the PRD merges) — the catch-all is for non-blocking, intentionally-deferred work.
- **HITL gate — plan only.** Do NOT write production code, run migrations, or open a PR. Stop at "agreed plan on the issue, ready for `/execute-issue`."

## Process

### 0.5 Locate the build context — base branch + in-flight work (do NOT plan from the default branch)
A child issue of a PRD is built against the **PRD's base branch**, not the default branch. Sibling slices merge into that base branch as they ship (child PRs target the base branch, *not* the default branch), so **the default branch is stale for the code this slice builds on** — reading it produces outdated plans: planning a rename against types a sibling already renamed, missing a shared helper a sibling added, referencing a folder a sibling moved. This is a real, observed failure in the donor repo, not a hypothetical. **Before reading any code:**

1. **Find the PRD + its base branch.** Read the issue's *Parent PRD* link for the PRD number, then read the base branch off any child PR of that PRD:
   `gh pr list --repo <owner/repo> --state all --search "<PRD#>" --json number,baseRefName,state,title` — the recurring non-default `baseRefName` (typically `prd-<n>-<slug>`) is the PRD base branch. If every child targets the default branch, or the issue is standalone with no parent PRD, there is **no** base branch — plan from the default branch and say so.
2. **Fetch it and see what siblings already landed.** `git fetch origin <base>` then `git log --oneline <default-branch>..origin/<base>` — each commit is a merged sibling slice whose changes this slice **inherits**. Scan for anything touching the same files/types (a merged rename, a new shared type, a moved folder).
3. **Read the code AT the base ref, not the working tree.** Check out the base branch if the working tree is clean, or read targeted files at the ref without disturbing anything: `git show origin/<base>:<path>`. When verifying a type/name still exists (or its current spelling), grep the **base ref**, not the default branch.
4. **Note in-flight siblings.** `gh pr list --repo <owner/repo> --base <base> --state open --json number,title,headRefName` — open PRs are slices being built **right now**; their diffs may move the ground again before this slice lands. Flag any that overlap this slice's files as a coordination risk in the plan.
5. **Inherit the plans of expanded-but-unbuilt siblings.** In the run-ahead flow (human expands children while the coder builds), upstream siblings — especially this issue's blockers — may be `📋 planned` but not yet merged. Read their `## Implementation plan (agreed via /expand-issue)` sections and walkthrough comments, and **plan as if that work will land exactly as agreed**: build on the interfaces, types, and slice layouts those plans commit to, as confidently as if they were already on the base branch. Name each inherited element in the plan ("inherits `IWidgetStore` from #<sibling>'s plan"). If this slice needs something an upstream plan doesn't provide, flag the gap NOW — to the user, and as a note on the upstream issue — rather than letting the coder discover it at execution.

State the base branch (or "none — default branch") **explicitly at the top of your analysis**, so the reader knows which tree the plan was built against. If a sibling has moved the ground since the issue was written, **reconcile it in the plan** (fix stale name references, note the new dependency) rather than planning against the stale shape.

### 1. Load the issue + the gospel
- `gh issue view <n> --repo <owner/repo> --comments`.
- From the body, **follow every hard-link**: read the cited design-doc section(s) (spec cards, contract docs, ADRs) and the named golden fixture(s). If a hard-link points at a doc the repo has since retired (per its ADR record), treat the code + tests + ADRs as the replacement source and note the dead link.
- Read the **actual code** the slice will touch **at the PRD base ref from Step 0.5** (not the default branch): the target project files, the existing slices to match conventions, and — in brownfield repos with a declared legacy oracle (profile `legacy_oracle`) — the oracle source it ports. A name you read on the default branch may already be renamed/moved on the base branch by a merged sibling.
- If a gospel link is missing or stale, **stop and flag it** — an issue without its gospel link isn't ready to plan.
- **The consumer is part of the gospel** for any slice touching a contract another system calls (route, verb, request/response shape, required field, resource decomposition). Read the parent PRD's consumer-reality section if it has one; if it's absent or the slice's specific surface isn't covered, probe the real consumer / legacy edge the profile's External Contracts section declares before drafting the approach. Pin claims to a route constant + verb, never a folder/component name. A shape the PRD asserts but no consumer citation supports is a **stale-gospel flag** — raise it with the user rather than planning against it.
- If the gospel itself already classifies something as a deliberate deferral, don't re-litigate it in the plan — cite it and route it to the deferred-work catch-all per the Principles above.

### 2. Draft the approach (in the coder lens's terms)
Produce a concrete plan covering:
- **Placement** — where the work lives per the repo's architecture doctrine (e.g., which feature slice / module, and its internal layout).
- **Deep module(s)** — the small interface(s) that hide the complexity, designed for testability (accept dependencies, return results, small surface — see the `codebase-design` skill).
- **Red-green test list** — the behaviors to test, in order (one test → one implementation), through public interfaces; what's mocked (boundaries only — HTTP, DB, time).
- **Parity plan** (brownfield, when the profile declares a legacy oracle) — which golden fixture / truth-table is the binding gate, and how it's generated from the oracle.
- **Open decisions** — the genuine forks that need the user (typically 1–4), each with *your recommendation* + the trade-off.

**For every element above, state the WHY** — the problem it solves and what fails without it — so the plan *teaches* rather than just specifies. Where a pattern, library mechanic, or failure mode is non-obvious, explain it plainly before relying on it.

### 2.5 Teach it — zoom out → zoom in (the briefing)
Deliver a structured, senior-to-senior briefing whose **exit bar is: the user is ready to review the PR / maintain the slice as if they wrote it** — not merely "informed." Cover, grounded in real code (`file:line`), the gospel, and (where one exists) the legacy-oracle source it ports:

- **Lead with a layman rundown (always).** Before any technical zoom, open with a plain-language headline — *what it is · the problem it solves · in → out · a one-line analogy* — **and a plain-language traced cross-module example**: follow ONE concrete token/value as it flows between modules, naming each handoff in everyday terms (e.g. *"a blank arrives → this slice hands the address to the lookup clerk → gets back the real name → drops it in"*). Cast already-built collaborators in plain roles ("lookup clerk", "safe calculator") and show what this slice hands each and gets back, so a non-specialist sees how the pieces snap together. A senior peer skims it in 20s; a layman *gets* it. This is non-negotiable and comes first.
- **Zoom out — place & purpose.** The slice's job in one sentence. What calls it / what it calls — its node in the pipeline. The behavior it mirrors (if ported) and the contract it must honor. Its boundaries (in scope vs. explicitly not). *Why it exists at all.*
- **Zoom in — how it actually works.** Trace ONE concrete input → output end to end (a real worked example, not hand-waving). Name the key abstraction(s) and explain *why this shape* — what complexity the deep module hides, and what its interface deliberately does NOT expose. Walk the non-obvious mechanics (the parse, the normalization, the order of operations) where they carry the behavior.
- **Sharp edges (what makes it reviewable).** The failure modes, parity traps, and easy-to-get-wrong spots; the invariants that MUST hold; the gotchas (encoding/formatting quirks, spec-variant ambiguities, the security boundary, concurrency, …). State them as review heuristics: *"if you see X in the PR, that's a bug; the risky file is Y."*
- **Change-impact model (what makes it maintainable).** How you'd extend/modify it; what ripples; safe vs. dangerous edits; where the seams are; what the tests catch vs. miss.
- **Check understanding.** Close with 2–3 targeted prompts (*"given input X, what's the output?"*, *"where would a parity bug most likely hide?"*), or invite the user to probe until review-ready. This step is done when the user can review it — not when they say "ok."

### 3. Resolve the decisions with the user (one at a time)
Present the proposal, then walk the open decisions one branch at a time. For any spike/architectural call: scope the spike (what it tests, against which fixtures), get approval, run it, bring back the evidence, let the user decide. Iterate until the approach is agreed. Keep the user's answers; don't silently re-decide them.

### 4. Persist the agreed plan onto the issue
Append an `## Implementation plan (agreed via /expand-issue)` section to the issue **body** so the coder and any future agent inherit it (the tracker is the durable state — this is what makes switching agents work):
```bash
gh issue view <n> --repo <owner/repo> --json body -q .body > <scratch>/body.md
cat >> <scratch>/body.md <<'EOF'

## Implementation plan (agreed via /expand-issue)
<layout · deep-module interfaces · ordered red-green test list · parity gate (if oracle-bound) · resolved decisions with rationale>
EOF
gh issue edit <n> --repo <owner/repo> --body-file <scratch>/body.md
```
Capture: the layout, the deep-module interface(s), the ordered red-green test list, the binding parity gate (if any), and each resolved decision **with its rationale**. Do NOT alter the Acceptance Criteria except to *sharpen* them when a decision changed them — and note the change explicitly.

### 4.5 Post the briefing as a walkthrough comment
The Step 2.5 briefing is the most maintainer-valuable artifact this skill produces — a plain-English, *why-first* explanation of the slice. Persist it **as a separate issue comment** (not in the body — the body holds the terse implementation plan; the comment holds the teaching narrative). The walkthrough comment is the slice's **temporary deep-dive artifact** — a point-in-time planning product, correct as of writing, with no maintenance contract. Per the lean doc canon (`doctrine/documentation-first.md`), it is never lifted into a standing doc; durable knowledge distils into ADRs / the glossary / the architecture map only.

- Render the **final, agreed** briefing (after the user's decisions are resolved — fold the resolved forks into the prose, drop the open-question framing). **Open with a plain-language headline rundown** — *what it is · the problem it solves · in → out · a one-line analogy*, readable by a non-specialist — **including a plain-language traced cross-module example** (one concrete token followed through the modules, each handoff named in everyday terms, already-built collaborators cast in plain roles) — **then** descend into the technical worked example re-traced with `file:line`, the sharp edges as review heuristics, and the change-impact model. Lead accessible; go deep after; never open cold with the interface. Ground every claim in real code (`file:line`) and the gospel.
- **First line MUST be the exact marker** `<!-- expand-issue:walkthrough -->` so the coder can find it deterministically.
- Post it: `gh issue comment <n> --repo <owner/repo> --body-file <scratch>`. If a walkthrough comment already exists (re-planning), edit it rather than posting a duplicate.

### 4.7 Emit the build-status board (chat output — every run)
After persisting, render a **"where are we" table of the remaining (non-shipped) issues** in chat, so the user has a clear visual of the build frontier. **Ordering is two-key, in this priority:**
1. **Coherence first** — order rows by the **runtime pipeline data-flow** (derive it from the repo's architecture/concept map — per the profile — and the PRD's suggested build order), **NOT by issue number**. The reader must see "we just did X → the next logical step is Y." Show branches off the main line indented (e.g. `↳ gap detection`). A module that runs early but builds late stays in its **runtime** position with a `*` note.
2. **Blockers second** — within that flow, annotate readiness so the eye lands on what's actually pickable now.

**Derive status from the repo, never memory:** closed issue / merged PR → `🚢 shipped`; issue carries an `<!-- expand-issue:walkthrough -->` comment → `📋 planned` (coder-ready); else `⚪ not started`. Read each issue's `Blocked by` lines for the blocker column.

Columns: **Stage (pipeline order) · Issue · Status · Blocked by · Next**, where **Next** ∈ { `🟢 coder builds now` (📋 + unblocked) · `🟡 expand next` (⚪ + unblocked) · `⏳ blocked/downstream` }. Put a **◀ you are here** marker on the last-shipped stage, and flag the just-planned issue. Close with a one-line takeaway: `coder → <issues> · expand next → <issues>`.

### 5. Hand off
Declare the issue planned and ready, then stop:
> #<n> planned — implementation plan on the issue body, plain-English walkthrough posted as a comment, build-status board emitted (Step 4.7). Ready: run `/execute-issue <prd>` to implement (coder builds to this plan + ACs), then `/review-pr <pr>`. Run ahead: `/expand-issue <next>` (the first 🟡 on the board) while the coder builds — its plan will inherit this one (Step 0.5, point 5).

Do NOT implement. Do NOT merge. The HITL gate ends here — the human owns planning; the agents own implementation + review.
